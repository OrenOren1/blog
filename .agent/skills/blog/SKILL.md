# Blog Writing Skill (Astro, agent-agnostic)

Use this skill whenever the user asks to draft, refine, or publish-ready a blog post in this repo.

This skill is intentionally tool-neutral so it can be used by OpenCode, Claude Code, Cursor, Copilot agents, or any markdown-capable assistant.

## Goals

1. Produce portfolio-ready markdown under `src/content/blog/`.
2. Match Oren's writing style: practical, first-person, experience-backed, DevOps-focused.
3. Keep read length intentional (default 5-8 minutes).
4. Share real-world experiences from Kubernetes, GitOps, AI/ML integration projects.

## Canonical Paths

- Blog posts: `src/content/blog/`
- Kubernetes posts: `src/content/blog/kubernetes/`
- GitOps posts: `src/content/blog/gitops/`
- AI/ML posts: `src/content/blog/ai-ml/`
- Platform engineering: `src/content/blog/platform/`

## Post Type Routing

Choose one route before writing:

1. `general` post -> `src/content/blog/YYYY-MM-DD-topic-slug.md`
2. `kubernetes` post -> `src/content/blog/kubernetes/topic-slug.md`
3. `gitops` post -> `src/content/blog/gitops/topic-slug.md`
4. `ai-ml` post -> `src/content/blog/ai-ml/topic-slug.md`
5. `platform` post -> `src/content/blog/platform/topic-slug.md`

## Frontmatter Contract (Astro content schema)

Aligned to `src/content.config.ts`.

Required:

- `title: string`
- `date: ISO date`

Strongly recommended for publish quality:

- `meta_title: string`
- `description: string`
- `image: string` (absolute URL or `/images/...`)
- `categories: string[]`
- `tags: string[]`
- `author: "Oren Sultan"`
- `draft: boolean`

Optional supported fields:

- `aliases: string[]`
- `featured: boolean`
- `unlisted: boolean`
- `source: general | kubernetes | gitops | ai-ml | platform`

Notes:

- Keep `draft: true` while drafting unless explicitly asked to publish.
- `minutesRead` is auto-generated by the remark pipeline. Never set it in frontmatter.

## Voice Profile (Write Like Oren)

Target style:

- First-person practitioner voice.
- Lead with operational reality, not abstract theory.
- Strong hook and clear TL;DR.
- Opinionated but specific: call tradeoffs, constraints, and failure modes.
- Concrete examples from real projects (LinearB, Terra.security, ZoomInfo, Beti).
- End with practical next step or action prompt.

Avoid:

- Generic AI fluff.
- Empty hype language.
- Long academic detours without practical outcome.
- Code dumps with no narrative context.

```markdown
  - "..."

  - "..."

author: "Oren Sultan"
source: "general"
---

## TL;DR

2-4 sharp sentences with practical value.

## Introduction

Hook from real engineering pain, tradeoff, or decision.

## Section 1

Context and framing.

## Section 2

Mechanics, patterns, commands, or architecture.

## Section 3

Operational guidance, failure modes, and recommendations.

## Conclusion

Clear close with what to do next.

## Resources

- Links to docs, repos, and related posts.
```

## Read-Length Guardrails

Default target:

- 5-8 minutes (roughly 900-1600 words).

If a draft exceeds target, tighten repetition first before removing core technical substance.

## Execution Checklist for Any Agent

Before marking done:

- Path matches post type.
- Frontmatter fields are valid and complete.
- TL;DR exists.
- Read length target is respected.
- Links are real and relevant.
- Draft status reflects user intent.
